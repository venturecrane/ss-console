# Adapter: Next.js page component (Tailwind v4)

Target stack: Next.js 16 + React 19 + Tailwind v4 (`@theme` in `globals.css`, no `tailwind.config`) + Clerk auth + Vercel deploy.

Matches: ke-console, dc-console.

## Target file conventions

**Component:** `src/components/<area>/<ComponentName>.tsx`

Where `<area>` is the surface class in lowercase kebab-case and `<ComponentName>` is PascalCase derived from the surface name. Before choosing a path, read the venture's existing `src/components/` tree — ke-console uses `src/components/ui/`, dc-console organizes by domain (`editor/`, `layout/`, `project/`, `sources/`). Match whatever convention already exists.

**Preview route:** `src/app/design-preview/<surface-name>/page.tsx` (app-router convention; no underscore prefix — Next.js does not auto-exclude underscore routes).

**Fixture:** `src/app/design-preview/<surface-name>/fixture.json` (co-located with the preview page).

  // Match existing venture conventions for type imports.
}

export function <ComponentName>({ /* destructured props */ }: Props) {
  return (
    // Component markup. Tailwind v4 classes referencing @theme tokens.
    // Compose with existing venture components — see component source
    // passed in the prompt. Respect the nav-contract block's chrome

Server component by default (no `'use client'`). Add `'use client'` only when the component uses browser APIs, event handlers, or React hooks that require the client runtime. Do not add it preemptively.

## What the generator must do

**Before writing:**

- Read every existing `.tsx` file under `src/components/**` (passed as prompt context). Identify which ones are reusable for this surface (shell wrappers, headers, cards, badges, status indicators, etc.).
- Import and use them with their actual prop signatures. Do NOT invent parallel components.
- Match the venture's existing type imports — ke-console defines shared API types in `src/lib/api.ts`; dc-console uses hook return types from `src/hooks/`. Use those, don't redefine.

**Component rules:**

- TypeScript strict mode. All props typed via an explicit `Props` interface. No `any`.
- No data fetching (`fetch`, DB queries, `useEffect`-based API calls) inside the component. The page or server component that renders it owns loading.
- No Clerk auth calls (`useUser`, `useAuth`, `auth()`) inside the generated component. Accept user data as props if needed.
- Type imports use the venture's path aliases (check `tsconfig.json` for `@/` alias root).

**Markup rules:**

- Tailwind v4 classes referencing the venture's `@theme` tokens. No arbitrary hex colors — ke-console uses `text-ke-*` / `bg-ke-*` semantic classes; dc-console uses `var(--dc-color-*)` CSS custom properties referenced via utility classes. Match whichever pattern the venture uses.
- Respect the nav-contract block injected into the prompt.
- Semantic HTML: `<main>`, `<section>`, `<article>`, `<header>`. One `<h1>` per component tree if this is the page's main content; `<h2>` for section headers; never skip levels.
- ARIA landmarks and accessible labels. Minimum touch target 44px for interactive elements (both ventures enforce this in existing components).

**Styling rules:**

- Color tokens: ke-console uses Tailwind semantic classes bound to `@theme` variables (e.g., `bg-ke-surface`, `text-ke-muted`, `border-ke-border`). dc-console references CSS variables directly in class values (e.g., `text-[var(--dc-color-text-muted)]`). Read `globals.css` before picking classes — don't mix patterns.
- No `@apply` directives (Tailwind v4 discourages them; use utility classes directly).
- No arbitrary hex or rgb values inline.
- Spacing and typography: follow the venture's declared scale from `globals.css` `@theme` block.

## Preview route template

<!-- prettier-ignore -->
```tsx
// src/app/design-preview/<surface-name>/page.tsx
// Dev-only preview for /product-design output. Not served in production.

import { notFound } from 'next/navigation'
import fixtureData from './fixture.json'
import { <ComponentName> } from '@/components/<area>/<ComponentName>'

export default function DesignPreview<ComponentName>Page() {
  if (process.env.NODE_ENV !== 'development') {
    notFound()
  }

  return (
    <html lang="en">
      <body>
        <<ComponentName> {...fixtureData} />
      </body>
    </html>
  )
}
```

The `notFound()` guard is the production safety mechanism. Unlike Astro's `_`-prefixed routes, Next.js app-router does not auto-exclude any path from builds — the runtime check is mandatory. `notFound()` returns a 404 and halts rendering; no need for a redirect.

The preview page lives outside Clerk middleware-protected paths by design — it does not need auth because generated components accept data as props, not from auth context.

## Fixture file template

```json
{
  "propName": "propValue",
  "listProp": [{ "id": "fixture-1", "name": "Sample", "status": "active" }],
  "nestedProp": { "field": "value" }
}
```

Shape matches exactly what the shipped page's data source returns. Synthetic but realistic: real-looking names, plausible values, at least one empty-state variant per fixture surface, long text fields that exercise wrapping.

## What this adapter refuses to do

- Emit `.astro` files — this adapter is for `.tsx` only
- Add `'use client'` to components that have no client-side behavior (hooks, event handlers, browser APIs)
- Call Clerk auth hooks (`useUser`, `useAuth`, `currentUser`) inside generated components — auth is the page's concern, not the component's
- Import from `src/app/` into `src/components/` (components don't depend on pages)
- Use `@apply` Tailwind directives
- Inline `style={{}}` blocks except for computed values genuinely inexpressible in utility classes

## ke-console specifics

- App root: `app/src/` (monorepo layout — the Next.js app lives under `app/`, not at repo root).
- Design tokens: `app/src/app/globals.css`, Tailwind v4 `@theme` block. Token prefix is `ke-*` (`bg-ke-surface`, `text-ke-primary`, `text-ke-muted`, `border-ke-border`, `text-ke-accent`, etc.).
- No middleware.ts in `app/src/` — auth is enforced via Clerk's `auth()` server function called directly in server pages. Generated components receive user/family data as props; they never call auth themselves.
- Existing UI components in `app/src/components/ui/`: `Card`, `CardHeader`, `CardBody`, `Button`, `Badge`, `StatusBadge`, `SectionHeader`, `EmptyState`, `Alert`, `Input`, `Select`, `Textarea`, `UpgradeModal`. Reuse these — do not reimplement.
- Type imports: `app/src/lib/api.ts` defines `Expense`, `MeResponse`, `BalanceResponse`, and related API shapes.
- Vercel deploy; no Cloudflare runtime constraints.

## dc-console specifics

- App root: `web/src/` (monorepo layout — the Next.js app lives under `web/`).
- Design tokens: `web/src/app/globals.css` (or `web/src/app/styles/`). Token prefix is `--dc-color-*`, `--dc-radius-*`, referenced in JSX as `text-[var(--dc-color-text-primary)]`, `bg-[var(--dc-color-surface-primary)]`, `rounded-[var(--dc-radius-md)]`, etc.
- Auth: `clerkMiddleware` in `web/src/middleware.ts` protects all routes not in the public matcher. Protected routes live under `src/app/(protected)/`. Design preview routes at `src/app/design-preview/` fall outside `(protected)` and outside the Clerk public matcher — they are open to the runtime `NODE_ENV` check with no Clerk dependency needed.
- Existing components: `layout/` (AppHeader, Sidebar, WorkspaceShell), `editor/` (EditorPanel, EditorToolbar, EditorSidebar, etc.), `project/` (ProjectSwitcher, ExportMenu, dialogs), `sources/` (SourcesPanel, SourceItem, etc.), `book/` (BookOutline, ChapterCard). Read the relevant area before generating — dc-console components are complex and client-heavy; respect the `'use client'` boundary decisions already made.
- Type imports: hook return types (`useProjectActions`, `useBackup`, etc.) in `web/src/hooks/`. Don't duplicate these shapes.
- Vercel deploy; no Cloudflare runtime constraints.
