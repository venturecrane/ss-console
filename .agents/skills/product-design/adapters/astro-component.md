# Adapter: Astro component (Tailwind v4)

Target stack: Astro 5 + React 19 islands + Tailwind v4 (`@theme` in `global.css`, no `tailwind.config`).

Matches: ss-console, smd-console, anything using the same stack.

## Target file conventions

**Component:** `src/components/<area>/<ComponentName>.astro`

Where `<area>` is the surface class in lowercase kebab-case (`portal`, `admin`, `public`, `auth`) and `<ComponentName>` is PascalCase derived from the surface name (`portal-quotes-detail` → `QuoteDetail`; `portal-home` → `PortalHomeDashboard`).

**Preview route:** `src/pages/design-preview/<surface-name>.astro` (surface-name in kebab-case matching the `--surface` argument).

**Fixture:** `src/pages/design-preview/<surface-name>.fixture.json`.

## What the generator must do

**Before writing:**

- Read every existing `.astro` file under `src/components/**` (passed as prompt context). Identify which ones are reusable for this surface (header chrome, tab navigation, list rows, card wrappers, status pills, money display, etc.)
- Import and use them with their actual prop signatures. Do NOT invent parallel components.
- Match the venture's existing type imports (e.g., ss-console may define `Quote`, `Invoice` types somewhere — use those, don't redefine).

**Frontmatter rules:**

- TypeScript strict mode. All props typed via an explicit `Props` interface.
- No data fetching (`fetch`, D1 queries, session access). The page owns that.
- Type imports use the venture's path aliases (check `tsconfig.json` / `astro.config.*` for aliases).

**Markup rules:**

- Tailwind v4 classes referencing the venture's `@theme` tokens. No arbitrary hex colors inline — use the semantic classes declared in the `@theme` block.
- Respect the nav-contract block injected into the prompt. If the block says "no footer on authenticated surfaces," don't emit a footer.
- Use semantic HTML: `<main>`, `<section>`, `<nav>` (only when the nav-contract allows it), `<article>`, `<header>` (via PortalHeader, don't hand-roll).
- Accessibility: proper heading hierarchy (one `<h1>` per component tree if this component is the page's main content; `<h2>` for section headers; never skip levels).
- Skip-to-main link, ARIA landmarks — unless the parent page already provides them (PortalHeader typically does).

**Styling rules:**

- Spacing: `.design/DESIGN.md` or the `@theme` block defines spacing scale. Use `p-4`, `gap-3`, etc. matching that scale, not arbitrary values.
- Color tokens: reference via Tailwind classes bound to `@theme` variables (e.g., `bg-primary`, `text-muted` if declared). Don't hand-type `#1E40AF`.
- Typography: use the venture's declared scale (display / title / heading / body-lg / body / caption / label — see ss-console's `global.css` for an example).
- Roundness: `.design/DESIGN.md` or `@theme` declares. Follow it (badge pill 9999px, card 12px, button 8px on ss-console).

## Preview route template

<!-- prettier-ignore -->
```astro
---
// src/pages/design-preview/<surface-name>.astro
// Dev-only preview for /product-design output. Not served in production.

import fixtureData from './<surface-name>.fixture.json'
import <ComponentName> from '@/components/<area>/<ComponentName>.astro'

// Production safety: this route must not serve in prod builds
if (!import.meta.env.DEV) {
  return Astro.redirect('/404')
}
---

<html lang="en">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1" />
  <title>[preview] <ComponentName></title>
  <link rel="stylesheet" href="/src/styles/global.css" />
</head>
<body>
  <<ComponentName> {...fixtureData} />
</body>
</html>
```

The `import.meta.env.DEV` check is belt-and-suspenders — Astro excludes `_`-prefixed routes from production builds, but the runtime guard catches any deployment misconfiguration.

## Fixture file template

```json
{
  "propName": "propValue",
  "listProp": [{ "id": "fixture-1", "name": "Sample", "status": "active" }],
  "nestedProp": { "field": "value" }
}
```

The shape matches exactly what the shipped page's data fetch returns for this surface. Hand-authored for gate 0 — ~20 lines per surface. Data is synthetic but realistic (real-looking names, plausible values, edge cases represented: empty states should have one fixture surface each, long text fields should exercise wrapping, etc.).

## What this adapter refuses to do

- Emit React component files (`.tsx`) when the surface's existing chrome is `.astro`
- Add `client:*` directives to static components (islands only when interactivity is genuinely required)
- Import from `src/pages/` into components (components don't depend on pages)
- Use `@apply` Tailwind directives (Tailwind v4 discourages them; use utility classes directly)
- Inline style blocks except when absolutely necessary (e.g., a computed animation that can't be expressed in utility classes)

## ss-console specifics

- Design tokens live in `src/styles/global.css` in a Tailwind v4 `@theme` block. Read it as part of prompt assembly.
- Existing portal components under `src/components/portal/`: `PortalHeader`, `PortalTabs`, `PortalListItem`, `ActionCard`, `StatusPill`, `ConsultantBlock`, `TimelineEntry`, `MoneyDisplay`, `ArtifactChip`. Reuse these — do not reimplement.
- `PortalTabs` is the navigation pattern (v3.1 persistent-tabs migration). Include it on every authenticated surface's top chrome along with `PortalHeader`.
- Type imports: check `src/lib/` for data types (`Quote`, `Invoice`, `Document`, etc.).
- Cloudflare Pages deploy target; no server runtime beyond what Astro SSR supports.
